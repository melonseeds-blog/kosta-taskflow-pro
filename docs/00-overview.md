## 읽는 순서

> **00 → 01 → 02 → 03 → 04 → 05**

- 위에서 아래로 갈수록 **추상 → 구체**로 좁혀진다.
- 앞 문서의 내용이 뒤 문서의 전제가 된다. (예: `02-specs`의 기능을 정해야 `03-design`의 API를 그릴 수 있다.)
- 새로 합류한 사람·다음 세션의 Claude는 **반드시 이 순서대로** 읽고 컨텍스트를 잡는다.

---

## 왜 이렇게 6개로 나누는가 — 관심사 분리 (Separation of Concerns)

문서 하나에 모든 것을 넣으면 **변경 비용**과 **탐색 비용**이 동시에 커진다. 그래서 각 문서가 **한 가지 질문**만 책임지도록 나눈다.

| 문서 | 답하는 질문 | 관심사 |
|---|---|---|
| `01-product` | **왜** 만드는가 / **누구**를 위해? | 제품 가치 |
| `02-specs` | **무엇을** 만드는가? | 기능 범위 |
| `03-design` | **어떻게** 만드는가? | 기술적 구현 |
| `04-tasks` | **언제 / 어떤 순서로** 만드는가? | 실행 계획 |
| `05-conventions` | **어떤 규칙**으로 만드는가? | 협업 규약 |

### 분리로 얻는 이점

1. **변경의 격리**
   디자인(03)이 바뀌어도 제품 정의(01)는 영향받지 않는다. 한 문서를 고쳐도 다른 문서가 깨지지 않는다.

2. **탐색의 효율**
   "API 스펙이 뭐였지?" → 곧장 `03-design`으로 간다. 한 문서를 위에서 아래까지 스캔할 필요가 없다.

3. **역할별 진입점**
   기획자는 `01~02`, 엔지니어는 `03~05`, PM은 `04`를 우선 본다. 모두가 같은 문서를 들고 다투지 않는다.

4. **추상화 단계 강제**
   "왜"가 정해지지 않은 채 "어떻게"를 정하는 실수를 막는다. 순서 자체가 사고의 가드레일이 된다.

5. **세션 간 컨텍스트 복원**
   Claude가 다음 세션에서 이 6개를 순서대로 읽으면, 인간 협업자가 며칠 만에 합류했을 때와 동일한 상태로 복귀한다.
